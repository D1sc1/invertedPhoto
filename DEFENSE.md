# Защита проекта «invertedPhoto»

Многопоточная инверсия (негатив) изображений на C++ с логированием и CI/CD.

---

## 1. Полное описание проекта

### 1.1. Назначение и постановка задачи
Программа загружает растровое изображение, инвертирует цвет каждого пикселя
(превращает фото в «негатив») и сохраняет результат. Главная инженерная идея —
**обработка выполняется параллельно в нескольких потоках**, после чего потоки
сливаются обратно в главный поток (`join`). Дополнительно ведётся **логирование**
и настроен **CI/CD** на GitHub Actions.

Инверсия пикселя — это формула `new = 255 - old` для каждого цветового канала
(R, G, B). Альфа-канал (прозрачность) не трогается.

### 1.2. Используемые технологии
- **Язык:** C++17.
- **Многопоточность:** стандартная библиотека `<thread>` (`std::thread`,
  `std::thread::hardware_concurrency`), `<mutex>`, `<chrono>`.
- **Работа с изображениями:** библиотеки `stb_image` / `stb_image_write`
  (public domain, заголовочные, без внешних зависимостей; лежат в `third_party/`).
- **Сборка:** CMake (≥ 3.15), генератор Ninja, компилятор GCC 13 (MinGW) / MSVC.
- **Тесты:** CTest + Python-скрипт проверки корректности.
- **CI/CD:** GitHub Actions (сборка на Linux и Windows, тесты, артефакты).
- **Файловая система:** `std::filesystem` (создание папок для вывода и логов).

### 1.3. Поддерживаемые форматы
- **Вход:** PNG, JPG, BMP, TGA, PSD, GIF, HDR, PIC, PNM.
- **Выход:** PNG, JPG, BMP, TGA (формат выбирается по расширению выходного файла).

### 1.5. Архитектура (поток выполнения)
```
                 ┌─────────────┐
   аргументы ──► │   main.cpp  │  разбор --threads/--log/-v, пути вход/выход
                 └──────┬──────┘
                        │
            ┌───────────▼───────────┐
            │   Image::load (stb)   │  читает пиксели в буфер (W*H*каналы байт)
            └───────────┬───────────┘
                        │
        ┌───────────────▼────────────────┐
        │  invertImageMultithreaded       │
        │  • делит пиксели на N чанков     │
        │  • запускает N std::thread       │   ← параллельная фаза
        │  • каждый поток: 255 - value     │
        │  • join() всех потоков           │   ← «слияние в один поток»
        └───────────────┬────────────────┘
                        │
            ┌───────────▼───────────┐
            │   Image::save (stb)   │  пишет результат по расширению
            └───────────────────────┘

   Параллельно всё пишется в лог (консоль + файл) через потокобезопасный Logger.
```

### 1.6. Как именно работает многопоточность (ключевой модуль)
Файл `src/inverter.cpp`, функция `invertImageMultithreaded`:

1. **Сколько потоков.** Если число не задано (`--threads 0` или флаг не указан),
   берётся `std::thread::hardware_concurrency()` (число логических ядер). Если ОС
   вернула 0 — берётся 1. Потоков не может быть больше, чем пикселей.

2. **Деление работы (декомпозиция данных).** Всего пикселей `P = W*H`. Они делятся
   на непрерывные диапазоны:
   - `base = P / N` — базовый размер чанка;
   - `remainder = P % N` — остаток; первые `remainder` потоков получают на 1 пиксель
     больше. Так нагрузка распределяется максимально равномерно (разница ≤ 1 пиксель).

3. **Параллельная фаза.** Для каждого чанка `[start, end)` создаётся `std::thread`,
   который вызывает `invertRange` и инвертирует свой диапазон:
   `px[c] = 255 - px[c]` для цветовых каналов, альфа не меняется.

4. **Слияние в один поток (`join`).** Главный поток в цикле вызывает `worker.join()`
   для каждого потока — ждёт завершения всех воркеров и «сливает» их обратно в себя.
   Только после этого изображение сохраняется.

5. **Замер времени.** Вокруг параллельной фазы стоит `std::chrono::steady_clock`,
   результат возвращается в структуре `InvertStats { threadsUsed, elapsedMs }`.

### 1.7. Почему не нужны блокировки на данных изображения
Каждый поток пишет в **свой непересекающийся диапазон байт** буфера. Два потока
никогда не трогают один и тот же байт, поэтому **гонки данных нет** и мьютексы для
самого изображения не нужны — это даёт максимальную параллельность. Видимость
результатов гарантируется самим `join()`: он создаёт отношение «happens-before»,
поэтому после join главный поток видит все изменения, сделанные воркерами.

Единственный общий ресурс — вывод в консоль и лог-файл; он защищён мьютексом
внутри `Logger`.

### 1.8. Логирование (`src/logger.cpp`)
- **Потокобезопасный синглтон.** Один экземпляр `Logger::instance()`; все записи
  сериализуются через `std::mutex`, поэтому строки из разных потоков не смешиваются.
- **Двойной вывод:** в консоль (stdout для INFO/DEBUG, stderr для WARN/ERROR) и в
  файл (с немедленным `flush`).
- **Уровни:** DEBUG, INFO, WARN, ERROR; порог задаётся (`--verbose` → DEBUG,
  `--quiet` → WARN).
- **Формат строки:** `[時間с миллисекундами] [УРОВЕНЬ] [tid идентификатор] сообщение`.
  Идентификатор потока (`std::this_thread::get_id()`) позволяет в логе наглядно
  видеть, как работают разные воркеры.
- **Кроссплатформенность:** время форматируется через `localtime_s` на Windows и
  `localtime_r` на POSIX.

### 1.9. Работа с изображением (`src/image.cpp`)
- Класс `Image` владеет буфером, выделенным `stb` (идиома **RAII**: память
  освобождается в деструкторе через `stbi_image_free`).
- Копирование запрещено, реализовано **перемещение** (move-конструктор и
  move-присваивание), чтобы исключить двойное освобождение.
- `load` читает все каналы как есть; `save` выбирает кодировщик по расширению.

### 1.10. Тестирование (CTest + round-trip)
Тест `roundtrip` (скрипт `scripts/verify_roundtrip.py`) проверяет корректность,
опираясь на математическое свойство инверсии: **двойная инверсия возвращает
оригинал**, значит `inv(inv(inv(x))) == inv(x)`.

Картинка инвертируется три раза. Так как PNG-кодировщик `stb` детерминирован
(одинаковые пиксели → одинаковые байты файла), PNG после **одной** инверсии должен
**побайтово** совпасть с PNG после **трёх** инверсий. Скрипт сравнивает файлы и
возвращает код успеха/ошибки, который ловит CTest.

### 1.11. CI/CD (`.github/workflows/ci.yml`)
Запускается на каждый push/PR в `main`. Шаги:
1. checkout репозитория и установка Python;
2. конфигурация и сборка проекта (**матрица: `ubuntu-latest` и `windows-latest`**);
3. запуск `ctest` (тест round-trip);
4. генерация тестовой картинки и её инверсия с подробными логами;
5. выгрузка **артефактов**: инвертированное изображение, лог-файл, исходная картинка
   (по одному набору на каждую ОС).

Это и есть «CI/CD»: каждое изменение автоматически собирается на двух платформах,
проверяется тестом и оставляет скачиваемые артефакты.

## 4. Вопросы и ответы (подготовка к защите)

### Многопоточность

**В: Что значит «слить в один поток»?**
О: Это операция `join`. Главный поток запускает рабочие потоки, а затем для каждого
вызывает `join()` — блокируется, пока поток не закончит, и «присоединяет» его
обратно. После джойна всех потоков продолжает работать только главный поток.

**В: Как вы делите работу между потоками?**
О: По пикселям. Всего `P = W*H` пикселей делятся на N непрерывных диапазонов.
`base = P/N`, остаток `P%N` раздаётся по одному пикселю первым потокам, чтобы
нагрузка отличалась максимум на один пиксель.

**В: Почему по пикселям, а не по строкам?**
О: Деление по пикселям даёт идеально равномерную нагрузку при любом размере. Деление
по строкам тоже возможно, но при числе строк, не кратном числу потоков, баланс хуже.
Оба варианта дают непрерывные участки памяти (хорошо для кэша).

**В: Нужны ли мьютексы при обработке пикселей? Есть ли гонки данных?**
О: Нет. Каждый поток пишет в свой непересекающийся диапазон байт, общих записей нет,
поэтому гонок данных нет и синхронизация не нужна. Это сознательное решение ради
скорости.

**В: А где тогда нужен мьютекс?**
О: Только в логгере — консоль и лог-файл это общий ресурс. Без мьютекса строки из
разных потоков смешивались бы.

**В: Как гарантируется, что главный поток увидит результаты воркеров?**
О: Через `join()`. Он устанавливает отношение «happens-before»: все записи воркера
гарантированно видны главному потоку после успешного join. Поэтому сохранение идёт
строго после джойна.

**В: Что такое false sharing и есть ли он здесь?**
О: Это когда разные потоки пишут в разные данные, попавшие в одну кэш-линию (64
байта), и процессор вынужден синхронизировать кэши. У меня чанки большие и
непрерывные, общими могут быть только байты на самых границах диапазонов — это
ничтожно мало и на производительность практически не влияет.

**В: Почему `std::thread`, а не OpenMP / `std::async` / пул потоков?**
О: `std::thread` — это чистый стандарт C++ без дополнительных зависимостей и флагов
компилятора, и он нагляднее показывает суть (создание, join). OpenMP проще писать,
но требует поддержки компилятором; пул потоков избыточен для одноразовой задачи;
`std::async` даёт меньше контроля над числом потоков.

**В: Какое ускорение даёт многопоточность?**
О: Задача «эмбарассингли параллельна», но упирается в пропускную способность памяти,
а не в вычисления (инверсия — это одна операция вычитания на байт). Поэтому ускорение
ограничено памятью, а не числом ядер. На маленьких картинках накладные расходы на
создание потоков могут даже превысить выигрыш — для них быстрее один поток.

**В: Что будет, если потоков больше, чем пикселей?**
О: Число потоков ограничивается числом пикселей, лишние не создаются.

**В: Что если `hardware_concurrency()` вернёт 0?**
О: По стандарту это допустимо (значит «не удалось определить»). В этом случае я
использую 1 поток.

### C++ и архитектура

**В: Как хранится изображение в памяти?**
О: Один непрерывный массив байт, построчно, с чередованием каналов:
`data[(y*width + x)*каналы + c]`. Значения 8-битные (0–255).

**В: Кто освобождает память изображения?**
О: Класс `Image` по идиоме RAII: буфер выделяет `stb`, а деструктор освобождает его
через `stbi_image_free`. Копирование запрещено, реализовано перемещение, чтобы не
было двойного освобождения.

**В: Какая сложность алгоритма?**
О: Линейная, `O(W * H * каналы)` — каждый байт обрабатывается ровно один раз.
По памяти — `O(1)` дополнительно (инверсия «на месте», in-place).

**В: Почему C++17?**
О: Нужны `std::filesystem` (создание папок для вывода/логов) и удобные возможности
языка; при этом C++17 широко поддерживается компиляторами.

**В: Как обрабатываются ошибки?**
О: `load`/`save` возвращают `bool` и пишут причину в лог; `main` возвращает код
возврата (0 — успех, 1 — ошибка загрузки/сохранения, 2 — ошибка аргументов).

### Обработка изображений

**В: Что именно делает инверсия и как насчёт альфа-канала?**
О: Для каждого цветового канала `новое = 255 - старое`. Альфа (прозрачность) не
инвертируется — иначе менялась бы прозрачность, а не цвет. В коде это учтено:
для форматов с альфой (2 и 4 канала) последний канал пропускается.

**В: Почему `stb`, и почему он лежит в репозитории?**
О: `stb` — минималистичная public-domain библиотека из одного заголовка, без
системы зависимостей. Я «вендорю» её (кладу в `third_party/`), чтобы сборка была
воспроизводимой и не зависела от сети и установленных пакетов.

### CI/CD и тесты

**В: Как вы проверяете, что инверсия работает правильно?**
О: Тестом round-trip. Двойная инверсия возвращает оригинал, поэтому одиночная и
тройная инверсии должны давать одинаковый результат. PNG-кодировщик детерминирован,
поэтому файлы сравниваются побайтово. Тест зарегистрирован в CTest и запускается в CI.

**В: Что делает CI/CD?**
О: На каждый push GitHub Actions собирает проект сразу на Linux и Windows, запускает
тест, прогоняет программу на сгенерированной картинке и сохраняет логи и результат
как артефакты. Это гарантирует, что код собирается и работает на обеих платформах.

**В: Зачем матрица из двух ОС?**
О: Чтобы поймать платформозависимые ошибки (пути, перевод строк, разные API времени).
У меня, например, время форматируется по-разному на Windows и POSIX.

### Общие / на «подумать»

**В: Как можно ускорить программу дальше?**
О: Векторизация (SIMD — обрабатывать по 16–32 байта за инструкцию), обработка на
GPU, или ускорение ввода-вывода. Но так как задача упирается в память, главный
резерв — пропускная способность памяти, а не вычисления.

**В: Какие узкие места?**
О: Чтение/запись файла (диск) и пропускная способность памяти. Сама инверсия очень
дешёвая.

**В: Что бы вы улучшили в проекте?**
О: Добавить пул потоков для пакетной обработки многих файлов, поддержку других
операций (яркость, оттенки серого), прогресс-бар, бенчмарк-режим, и больше тестов
(в т.ч. на изображениях с альфа-каналом).

В проекте реализована законченная утилита обработки изображений, демонстрирующая:
- корректную **многопоточность** на `std::thread` с делением данных и `join`;
- грамотную работу с разделяемыми ресурсами (мьютекс только там, где он реально
  нужен — в логгере);
- **потокобезопасное логирование** с уровнями и идентификаторами потоков;
- современный **C++17** (RAII, move-семантика, `std::filesystem`, `std::chrono`);
- автоматизацию качества: **CMake + CTest + CI/CD** на двух платформах.
